ACTION BUTTON RECORD CONTRACT — V1

Role:
Action buttons are saved source-callers.
They do not define the full prompt/preset system.
They call saved source records such as prompts or mapped prompts and make them easier to use from pages like chat.

For phase 1:
- buttons are usable records
- chat page can use them
- button editor page creates/edits them
- deeper hotkey/widget/global execution systems come later

--------------------------------------------------
1. BUTTON SYSTEM MODEL
--------------------------------------------------

Use:
- /buttonlibrary/

Button records use the same general data model:
- .json = truth
- .md = mirror

Later:
- verified buttons become the safer preferred/allowed set for wider use
- for now, any button can be used if minimum fields are filled

Core model:

button library = saved button records
button editor page = creates/edits them
chat page = calls them
future widget/hotkey layer = executes them in broader ways

--------------------------------------------------
2. BUTTON TYPE ROUGH NAMES
--------------------------------------------------

Keep the rough names for now as long as they map clearly to behavior:

1. Raw Send Button
2. Window-Wrap Button
3. Custom-Input Template Button
4. Prompt-Slot Switcher Button

These names can be cleaned later if needed.

--------------------------------------------------
3. BUTTON TYPE BEHAVIOR
--------------------------------------------------

1. Raw Send Button
- has a prompt or prompt map attached
- pressing it sends produced raw output to clipboard
- primary target now = Debian Wayland clipboard
- use system-neutral fallback when practical
- later may support mouse-location or hotkey execution

2. Window-Wrap Button
- uses a map
- wraps either:
  - user input
  - or upload-window content
- depends on a guaranteed source window being attached

3. Custom-Input Template Button
- loads a map
- lets the user fill chosen custom input fields inside the button/widget area
- pressing apply saves temp values
- later uses those temp values each time until changed
- currently focused on headers/prompts, not arbitrary command execution

4. Prompt-Slot Switcher Button
- loads a map with multiple chosen prompt slots
- lets the user swap prompts per slot
- can toggle fields off temporarily
- has apply and send
- likely the most complex button page/type

--------------------------------------------------
4. BUTTON RECORD FIELDS
--------------------------------------------------

Phase 1 button record fields should include:

Core identity:
- button name
- button type
- status
- tags
- notes/description if useful

Source links:
- linked prompt
- linked prompt map
- linked source window if needed

Custom input support:
- adjustable/custom input fields if used
- temp values if used

Visual style:
- button background color
- button color
- text color
- border color
- border/style type if practical

Behavior/use fields:
- send/apply mode
- field or slot behavior relevant to the chosen button type

Minimum rule:
- if minimum required fields are filled, button can be usable
- verification/preference can come later

--------------------------------------------------
5. BUTTON PAGE / GROUP MODEL
--------------------------------------------------

Button pages/groups should save:
- list/order of buttons
- layout
- toggles

They should NOT save temp-state for phase 1.

This lets button pages act like:
- organized button groupings
- reusable layouts
- future widget-like control groups

--------------------------------------------------
6. TEMP VALUE RULE
--------------------------------------------------

For buttons that use custom adjustable values:
- temp values persist until manually changed
- one temp file per adjustable page is acceptable for phase 1

So:
- apply stores the temporary chosen values
- later uses continue using those values until changed manually

--------------------------------------------------
7. SEND / APPLY MEANING
--------------------------------------------------

Send meaning depends on context.

Upload-related send:
- copy input
- put it in display
- copy/temp-log until saved

User input related send:
- wrap text
- insert user_input into the correct slot
- for now, output goes to clipboard
- later may gain more output choices

Apply means:
- update the button/template/widget’s temporary chosen values or slot choices
- do not treat apply as final execution unless that specific button type needs it

--------------------------------------------------
8. VISUAL STYLE MODEL
--------------------------------------------------

Important visual layering rule:

These are not just flat buttons.
They are closer to small widgets with layered visual identity.

Phase 1 visual fields:
- button background
- button color
- text color
- border color/style

Meaning:
- button background = widget/background layer
- button color = actual clickable button layer

If needed in phase 1, colors can match for simplicity.
But the model should allow layering so the system does not become a monotone mess later.

Preferred color controls:
- popup color chooser / color wheel
- live visual feedback if practical

--------------------------------------------------
9. PAGE / EDITOR PLACEMENT
--------------------------------------------------

Button editing should live on its own page for now.

Chat page:
- uses buttons
- calls buttons
- does not become the full button editor system

This keeps:
- button creation/editing separate
- chat page focused on use
- future widget/hotkey extraction cleaner

--------------------------------------------------
10. RESIZE / SCROLL EXPECTATION
--------------------------------------------------

The future button page/editor should:
- resize without breaking badly
- keep important controls visible as long as practical
- become scrollable when shortened
- avoid collapsing into unusable clutter

Do not complicate this phase with global scroll-lag fixes across all pages.
Shared resize/scroll cleanup can happen later as one pass.

--------------------------------------------------
11. STATUS / USABILITY RULE
--------------------------------------------------

As with prompts/maps:
- status controls trust/preference
- not basic usability

So:
- active = verified/current/preferred
- archived = older but still valid/readable
- draft/review/copied/tuning can still be usable if minimum extractable fields exist

This applies to button records too.

--------------------------------------------------
12. FUTURE BOUNDARY
--------------------------------------------------

Do NOT do these yet:
- global hotkey execution system
- KDE/Wayland wrapper layer
- full widget execution routing
- advanced multi-output engines
- overly clever command systems

Safe rule:
prove the pattern in page-based button records first
extract the stable generic part later

--------------------------------------------------
13. SAFE SUMMARY
--------------------------------------------------

Action buttons are saved source-calling records.
They are stored in /buttonlibrary/ as .json truth + .md mirror.
They come in rough behavior-based types for now.
They support prompt/map linking, light custom inputs, visual identity, and grouped page layouts.
The button editor page owns creation/editing.
The chat page uses them.
Future hotkey/widget/global execution systems come later.
